One-time Setup

Do this once. Every new client project after this takes < 10 minutes. The goal is a GitHub template repo that is fully pre-wired.

Setup
Create the GitHub template repo
One repo to rule them all — every project clones from this
30 min ▾

Go to github.com → New Repository → Name it piaazo-web-template. After creation, go to Settings and check "Template repository".

  • Create repo: piaazo-web-template (private)
  • Enable "Template repository" in repo Settings
  • Push all files from the template package to this repo
  • Set branch protection on main: require CI to pass before merge
  • From now on: new client project = "Use this template" button
✓ After this, every project gets tests + CI/CD + monitoring pre-wired from day one.
Setup
Install and authenticate Vercel CLI
Required for headless deploys from the terminal and GitHub Actions
10 min ▾
terminal
# Install globally
npm install -g vercel@latest

# Login (opens browser)
vercel login

# Link your first project (run inside the repo)
vercel link

# This creates .vercel/project.json — get your IDs from there
cat .vercel/project.json
⚠ Add .vercel/ to .gitignore — project.json contains sensitive IDs
Setup
Add GitHub Secrets
The pipeline cannot deploy without these three secrets
5 min ▾

Go to: GitHub repo → Settings → Secrets and variables → Actions → New repository secret

  • VERCEL_TOKEN — get from vercel.com/account/tokens → Create Token
  • VERCEL_ORG_ID — from .vercel/project.json field orgId
  • VERCEL_PROJECT_ID — from .vercel/project.json field projectId
ℹ For multi-client (agency): use GitHub Organization Secrets so all repos inherit these automatically.

Build Phase

Starting a new client project from the template. The setup script handles all scaffolding automatically.

Build
Scaffold new client project
Use the template repo + run the setup script
10 min ▾
  • Go to github.com/YOUR_ORG/piaazo-web-template → "Use this template" → "Create a new repository"
  • Name it: client-[client-slug] (e.g. client-acme-dental)
  • Clone locally: git clone git@github.com:YOUR_ORG/client-name.git
  • Run the setup script: npm run setup
  • Script will prompt for client name, domain, stack, and Sentry — then wire everything
  • Add the 3 GitHub Secrets (VERCEL_TOKEN, ORG_ID, PROJECT_ID) for this new repo
✓ After npm run setup the project is ready: deps installed, .env.local created, Vercel linked, initial commit made.

QA & Testing

Three testing layers that run automatically. Manually run stress tests before any major client handoff.

Test
Playwright E2E tests (auto on every push)
Baseline spec covers every project. Add custom tests per client.
Auto ▾

The tests/e2e/baseline.spec.ts file runs automatically in GitHub Actions. It covers: page load, navigation, mobile viewport, no broken links, no console errors, image alt text, and form detection.

run tests locally
# Run all tests headless
npm test

# Run with UI debugger (great for writing new tests)
npm run test:ui

# Run against a live URL (e.g. Vercel preview)
BASE_URL=https://preview-abc123.vercel.app npm test
  • Add client-specific tests in tests/e2e/[client-name].spec.ts
  • Use the "Write Playwright tests" prompt in PROMPTS.md for AI-generated tests
  • Playwright reports saved as artifacts in GitHub Actions (download from Actions tab)
Test
Lighthouse CI (auto on every push)
Performance ≥ 85, Accessibility ≥ 90 — gates the deploy
Auto ▾

Lighthouse CI runs automatically in GitHub Actions. If scores drop below thresholds in lighthouserc.json, the deploy is blocked. Reports are uploaded to temporary public storage — the Actions job summary has the link.

run locally
# Build first, then audit
npm run build && npm run lhci

# Adjust thresholds in lighthouserc.json
Test
k6 stress test (run before client handoff)
Smoke → Load → Stress — run against the Vercel preview URL
Manual (15 min) ▾
stress test commands
# Smoke test (1 user — sanity check)
k6 run tests/stress/load.js

# Load test (50 users — normal traffic)
k6 run --env STAGE=load -e BASE_URL=https://preview.vercel.app tests/stress/load.js

# Stress test (100 users — find the breaking point)
k6 run --env STAGE=stress -e BASE_URL=https://preview.vercel.app tests/stress/load.js
  • Run smoke test: must pass before any client demo
  • Run load test: must pass before production handoff
  • Stress test: run once per project to establish baseline capacity
  • Results saved to tests/stress/results/summary.json
⚠ Run stress tests against Vercel preview URL — not production. Vercel's free tier may throttle.

Deploy

Fully automated after the one-time setup. Push to main → tests pass → production deploys. No dashboard clicks.

Deploy
What happens on every push to main
The full pipeline — automated, gated, no manual steps
~5 min total ▾
  • Build job — npm ci + next build. Artifact uploaded.
  • E2E job — Playwright runs on chromium + firefox + mobile iPhone 14. Any failure blocks deploy.
  • Lighthouse job — Performance/accessibility audit. Fail if below thresholds.
  • Deploy job (only if both above pass) — vercel build + vercel deploy --prod. Live URL in Actions summary.
✓ On PR (not main): same pipeline but deploys to a preview URL which is automatically posted as a PR comment for client review.
Deploy
Custom domain setup
Point client's domain to Vercel in 3 steps
15 min ▾
  • Vercel Dashboard → Project → Settings → Domains → Add domain
  • Copy the CNAME/A record Vercel shows you
  • Go to client's domain registrar (GoDaddy, Namecheap, etc.) → DNS → Add those records
  • SSL is automatic — Vercel provisions it within minutes
Deploy
Environment variables in production
Never hardcode secrets — set them in Vercel dashboard
5 min ▾

Vercel Dashboard → Project → Settings → Environment Variables. Add all variables from .env.local here, setting them for Production (and Preview/Development if needed).

⚠ Variables starting with NEXT_PUBLIC_ are exposed to the browser. Never put secret keys in these.

Monitor

Three monitoring layers. Set up once per project. You'll know about problems before the client does.

Monitor
Sentry — runtime error tracking
Catches JS errors, API failures, and unhandled exceptions in production
Auto after setup ▾
  • Set SENTRY_DSN in Vercel environment variables
  • Sentry auto-captures: unhandled exceptions, API route errors, Next.js SSR errors
  • Configure alerts: Sentry → Project → Alerts → "Alert me on new issues" (email)
  • Source maps uploaded automatically during build (by Sentry's Next.js plugin)
Monitor
BetterStack — uptime monitoring
Pings the site every 3 minutes. Emails + SMS if it goes down.
5 min per client ▾
  • betterstack.com → Uptime → New Monitor
  • URL: client's production domain. Check interval: 3 minutes.
  • Alert channels: email (yours + client's if appropriate)
  • Free tier: 10 monitors, 3-min checks — covers most agency work
  • Optionally add a public status page for the client (betterstack.com/status-page)
Monitor
Vercel Analytics + Speed Insights
Real user performance data — Core Web Vitals from actual visitors
2 min ▾
src/app/layout.tsx — add these imports
import { Analytics } from '@vercel/analytics/react';
import { SpeedInsights } from '@vercel/speed-insights/next';

// Add inside your root layout JSX:
<Analytics />
<SpeedInsights />

Then enable in Vercel Dashboard → Project → Analytics. Free with hobby plan. Shows LCP, FID, CLS by page, device, country.
